в карту страниц сайта

Search queries
become a sitemap of decisions

Агент берёт список поисковых запросов и описание бизнеса. Чистит мусор, понимает, что человек ищет (купить, узнать, сравнить), группирует похожие запросы — и для каждой группы предлагает страницу: создать новую, обновить существующую, написать статью, поднять FAQ или не брать в работу.

На выходе — не «список ключей», а таблица решений: группа → страница → действие. Для «Создать»-групп — сразу готовый title / H1 / description с проверкой длин и бриф копирайтеру с H2-структурой. Без ручного триажа после.

One pipeline turns raw search queries into a sitemap of actions. Filters noise, classifies intent, groups by meaning, and emits a decision per group: create, update, article, FAQ, or drop.

Output is not a keyword list but a decision table: cluster → page → action. For "Create" clusters — ready title / H1 / description with length checks and a copywriter brief with H2 structure. No manual triage downstream.

01

Как работает

Pipeline

Семь шагов превращают сырые запросы и описание бизнеса в карту решений по страницам. LLM включается только там, где правил недостаточно — определить намерение, назвать группу и собрать SEO-мету с брифом копирайтеру. Внешние источники (autosuggest, конкуренты) дают свежий вход; всё, что делается внутри pipeline после сбора — детерминированно.

Seven steps turn raw queries and a business spec into a page-decision map. LLM kicks in only where rules can't decide — intent classification, group labels and SEO meta + copy brief. External sources (autosuggest, competitors) provide fresh input; everything downstream of collection is deterministic.

01 СборCollect
Подсказки поисковика и парсинг конкурентов работают вживую. Wordstat, GSC, KeyCollector — пока заглушки, для реальных API нужны учётные данные и OAuth.
Autosuggest and competitor scraping are live. Wordstat, GSC, KeyCollector remain stubs — live wiring needs credentials and OAuth, not just a config flag.
02 ЧисткаClean
Сначала простые правила — нельзя бренды, юридические риски, прямой мусор. Это быстро и предсказуемо. Только после этого LLM решает, какое у запроса намерение. Дешевле и точнее, чем кидать всё в модель.
Rule filter first (brands, legal risks, garbage) — cheap and predictable. Only then LLM classifies intent. Cheaper and more accurate than feeding raw input to the model.
03 ГруппировкаCluster
Запросы переводим в числовые векторы (embeddings) через sentence-transformers. Большие группы дополнительно разбиваем по признакам — размер, материал, назначение. Похожие группы потом склеиваются.
Queries → vectors via sentence-transformers. Big clusters split by facets (size, material, use-case); near-duplicates merged downstream.
04 Подбор страницыMatch
Каждой группе ставится решение по странице: использовать существующую, создать новую категорию, написать гайд или не брать в работу. Важно: группа с коммерческим намерением никогда не уходит в блог — это закодировано в правилах матчинга.
Each cluster gets a page decision: reuse existing, create new category, write article, or drop. Hard rule: commercial intent never maps to /blog/.
05 Оценка приоритетаScore
Для каждой группы считаем приоритет по формуле (спрос × бизнес-ценность × шанс выйти в топ − сложность). Поле score_mode честно показывает, на чём оценка построена.
Each cluster gets a priority score (demand × business value × ranking opportunity − difficulty). score_mode states what the number is actually based on.
06 Поиск пробеловGap
Замкнутый цикл: новые группы сравниваются с тем, что уже есть на сайте и у конкурентов. Возвращает обновлённую таблицу — какие группы стоит добавить, какие расширить, какие убрать.
Closed loop: new clusters compared against current site + competitor pages. Returns an updated decision table — what to add, expand, drop.
07 SEO-метаSEO meta
Для каждой «Создать»-группы — title / H1 / description с проверкой длин и intent-aware промптом (для коммерческой — призыв к действию, для проблемной — вопросительная форма), плюс бриф копирайтеру: аудитория, H2-структура, тон, ключевые запросы. Файл на каждую группу в briefs/. Отключается флагом --no-seo.
For every "Create" cluster — title / H1 / description with length checks and intent-aware prompt (commercial → CTA, problem → interrogative title), plus a copywriter brief: audience, H2 structure, tone, target queries. One file per cluster in briefs/. Disabled via --no-seo.
02

Supervisor: когда повторить шаг

LangGraph supervisor

Весь пайплайн обёрнут в надстройку-supervisor на LangGraph. После критичных шагов она смотрит на реальные числа и решает — идти дальше или повторить с другими настройками. Это не «попробовать ещё раз», а два явных правила:

The whole pipeline is wrapped in a LangGraph state-graph. After critical steps, the supervisor inspects real metrics and decides whether to proceed or retry with adjusted parameters. Not a timer — two explicit guards:

yield_guard
Если после чистки осталось меньше 30% от собранного —
If <30% of collected queries survive cleaning —
список модификаторов узковат. Расширяем его и собираем заново. Одна повторная попытка, дальше пропускаем дефицит как факт.
expand modifier list and re-collect. One retry; if still low, log and proceed honestly.
size_guard
Если после группировки получилось меньше трёх кластеров —
If clustering yields fewer than three groups —
порог склейки был слишком жёсткий. Ослабляем его на +0.10 и группируем ещё раз. Одна повторная попытка.
similarity threshold too strict. Relax by +0.10 and re-cluster. One retry.

Журнал решений пишется в state["decisions"] — видно, какой guard сработал и почему. Реальный прогон на демо-домене: 252 секунды от seed-конфига до 16 финальных групп. На «хороших» входах guards не срабатывают — они страховка, не основной путь.

Decision log lives in state["decisions"] — every guard hit is recorded. Real demo run: 252 s from seed config to 16 final clusters. On clean input guards do not fire — they are insurance, not the default path.

Дальнейшие шаги

Roadmap

Что подключается следующим — и почему сейчас это честно видно в коде, а не закопано в копи.

What ships next — and why it stays honest in the code, not in the copy.

Wordstat подключён через CDP-парсинг живого wordstat.yandex.ru (Direct API упирается в регистрацию рекламодателя, что для тестового абсурдно — обошли через debug-порт браузера). 105 запросов получили реальный search_volume, score_mode для всех 20 кластеров — mixed (было demo). GSC и Ahrefs остаются stubs — для них нужны верифицированный сайт и платный ключ.

Wordstat is wired via CDP-scraping of the live wordstat.yandex.ru tab (Direct API requires advertiser registration, absurd for a test task — bypassed via the browser debug port). 105 queries got real search_volume, score_mode for all 20 clusters is now mixed (was demo). GSC and Ahrefs stay as stubs — verified site and paid keys respectively.

Различение кластеров по поисковой выдаче (SERP) — пока не реализовано. В брифе упоминается как future. Сейчас различаем группы по embeddings и label-склейке, не по выдаче Яндекса/Google.

SERP-based cluster differentiation is in the brief as future work. Currently clusters are distinguished by embeddings and label merge, not by live search results.

SEO-мета и бриф копирайтеру — собираются в шаге 07. Title / H1 / description с проверкой длин и intent-aware промптом, плюс markdown-бриф с H2-структурой в briefs/{slug}.md. Post-indexing verify (GSC → CTR/position через 14/30 дней после публикации) — следующий шаг, нужен OAuth.

SEO meta and copywriter brief land in step 07. Title / H1 / description with length checks and intent-aware prompts, plus a markdown brief with H2 structure under briefs/{slug}.md. Post-indexing verify (GSC → CTR/position 14/30 days after publication) is next; the blocker is OAuth.

В демо один домен (мебель). Универсальность под другие тематики не проверена — узкие seeds.yaml-конфиги переключают модификаторы и intent-правила, но валидации на e-commerce других вертикалей нет.

Single demo vertical (furniture). Generalisation across verticals is untested — modifier lists and intent rules swap by config, but no cross-domain validation yet.
